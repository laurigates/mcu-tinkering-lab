# PRD-012: Balancebot

**Status**: draft
**Created**: 2026-07-02
**Confidence**: 7/10

---

## Overview

Balancebot is a two-wheel self-balancing robot built from a Seeed Studio
XIAO RP2350, an MPU6050 IMU, and two DRV8825 stepper drivers turning
NEMA17 motors. It balances as an inverted pendulum: a 500 Hz control
loop reads the IMU, estimates pitch with a complementary filter, and
commands wheel acceleration through a PID cascade. Step pulses are
generated by RP2350 PIO state machines, so motor timing is jitter-free
regardless of what the CPU is doing.

The robot is tuned over USB CDC serial: a line-based CLI exposes the
PID gains live, streams telemetry as CSV for plotting, and persists
tuned gains to flash. Once tuned, the robot balances untethered.

This is the monorepo's first non-ESP32 firmware project. Alongside the
robot it introduces the Raspberry Pi Pico SDK toolchain to the repo:
containerized builds, shared justfile recipes, and a reusable CI
workflow that other RP2040/RP2350 projects can adopt.

## Problem

Self-balancing is the canonical hard-real-time control problem: it
needs a deterministic sensor→filter→PID→actuator loop, safe failure
modes, and live tuning to converge on stable gains. The repo has no
project exercising closed-loop motion control — the robocar drives
open-loop DC motors — and no project on non-Espressif silicon, so
there is no path for using the RP2350 boards already on hand. The
parts for a balancer (XIAO RP2350, MPU6050, DRV8825s, NEMA17s) are
available and otherwise idle.

## Goals

- Robot balances upright on two wheels, untethered, for minutes at a
  time and recovers from small pushes
- Deterministic 500 Hz control loop on a dedicated core; step-pulse
  generation offloaded to PIO state machines
- Live PID tuning and telemetry streaming over USB CDC; tuned gains
  persist across power cycles
- Safe by construction: drivers disabled through boot/reset, tilt
  cutoff, hardware watchdog
- Establish reusable Pico SDK build infrastructure (container, justfile
  recipes, CI workflow) for future RP2040/RP2350 projects

## Non-Goals

- No remote control or driving around in v1 (balance in place only)
- No wireless connectivity — the XIAO RP2350 has no radio; a radio
  module or companion board is a possible v2
- No wheel encoders — steppers are trusted not to skip within the
  commanded acceleration envelope
- No RISC-V (Hazard3) build — Cortex-M33 only in v1
- No OTA updates — firmware loads over USB (UF2 / picotool)

## User Stories

### Primary user — builder / tuner

- **US-01**: As a builder, I want a complete wiring guide (driver
  current limit, microstepping straps, IMU orientation, power
  topology), so that I can assemble the robot without reverse-
  engineering the firmware.
- **US-02**: As a tuner, I want to adjust PID gains live over a serial
  CLI while the robot balances, so that tuning is an interactive loop
  rather than an edit-flash-test cycle.
- **US-03**: As a tuner, I want telemetry streamed as CSV (angle, gyro
  rate, commanded rate, loop time), so that I can plot behaviour with
  any serial plotter.
- **US-04**: As a tuner, I want tuned gains saved to flash, so that the
  robot balances standalone after a power cycle.
- **US-05**: As an operator, I want the robot to disable its motors
  immediately when it falls past 45°, so that a fallen robot doesn't
  grind its wheels.

### Secondary user — developer

- **US-06**: As a developer, I want `just balancebot::build` to produce
  a UF2 inside a container, so that I don't install a local ARM
  toolchain or the Pico SDK.
- **US-07**: As a developer, I want CI to build the firmware on every
  PR touching the project, so that regressions are caught before merge.
- **US-08**: As a developer, I want the control-logic modules (PID,
  complementary filter, parameter CRC) unit-tested on the host in CI,
  so that control math is verified without hardware.
- **US-09**: As a developer starting the next RP2350 project, I want
  shared Pico SDK justfile recipes and a reusable CI workflow, so that
  the second project is scaffolding-free.

## Hardware

| Component | Specification |
|-----------|--------------|
| MCU | Seeed Studio XIAO RP2350 (dual Cortex-M33 @ 150 MHz, 520 KB SRAM, 2 MB flash, no radio) |
| IMU | MPU6050 (GY-521 breakout), I2C @ 400 kHz, data-ready INT |
| Motor drivers | 2× DRV8825 carrier boards, 1/8 microstepping (hardwired), STEP/DIR interface |
| Motors | 2× NEMA17 bipolar steppers (200 steps/rev; 1600 steps/rev at 1/8 µstep) |
| Wheels | ~80–100 mm diameter, directly on motor shafts |
| Motor power | 2–3S LiPo (7.4–11.1 V) → DRV8825 VMOT (100 µF bulk cap at each driver) |
| Logic power | Battery → 5 V buck converter → XIAO 5V pin (onboard LiPo charger is NOT used for motors) |
| Safety strap | 10 kΩ pull-up from shared DRV8825 nENABLE to 3V3 (drivers disabled while MCU boots/resets) |
| Frame | Two-deck chassis; MPU6050 mounted rigidly at axle height, X axis pointing forward |

## Features

### Phase 1 — Scaffold + build infrastructure

**FR-1.01**: Pico SDK project skeleton at
`packages/robotics/balancebot/` registered in the root justfile.

**FR-1.02**: Containerized build: `Dockerfile.pico` (pinned pico-sdk +
picotool), `pico-sdk` service in `docker-compose.yml`, shared recipes
in `tools/pico.just`.

**FR-1.03**: CI workflow `build-balancebot.yml` calling a reusable
`_ci-build-pico.yml`: builds the UF2, uploads artifacts, enforces a
flash-size gate.

**FR-1.04**: Boot firmware: status LED heartbeat + USB CDC version
banner, proving flash + monitor paths work.

### Phase 2 — IMU pipeline

**FR-2.01**: MPU6050 register driver over I2C1: WHO_AM_I check, DLPF
and range configuration, 14-byte burst read, data-ready interrupt.

**FR-2.02**: Boot-time gyro bias calibration (robot at rest).

**FR-2.03**: Complementary filter producing pitch angle and pitch rate
at 500 Hz.

**FR-2.04**: CLI skeleton with `stream` command emitting CSV telemetry.

### Phase 3 — Open-loop stepper drive

**FR-3.01**: PIO step-pulse generator: one state machine per motor,
variable rate 30–20 000 steps/s, ≥1.9 µs high time, glitch-free rate
updates, holds last rate between updates.

**FR-3.02**: DRV8825 abstraction: signed rate → DIR level + PIO period;
enable/disable via shared nENABLE; rate and acceleration clamps.

**FR-3.03**: `arm` / `disarm` / `rate <l|r> <steps/s>` CLI commands for
bench testing.

### Phase 4 — Closed-loop balancing

**FR-4.01**: 500 Hz control loop on core 1, paced by the IMU data-ready
interrupt with a timer fallback.

**FR-4.02**: Angle PID whose output is wheel acceleration, integrated
to step rate (clamped ~4000 steps/s).

**FR-4.03**: Safety: tilt cutoff (|pitch| > 45° → motors off + FAULT),
hardware watchdog fed only by the control loop, arm gesture (hold
upright ±1° for 3 s) or explicit `arm`.

**FR-4.04**: Lock-free core0↔core1 exchange: parameter queue in, seqlock
telemetry snapshot out.

### Phase 5 — Tuning shell + persistence

**FR-5.01**: Full CLI: `help stat get set stream cal arm disarm rate
save load`.

**FR-5.02**: Parameter store in the last 4 KB flash sector (magic +
version + CRC32), written via `flash_safe_execute()` only while
disarmed.

**FR-5.03**: Outer velocity loop (100 Hz): wheel-velocity feedback
trims the angle setpoint so the robot holds station instead of
drifting.

**FR-5.04**: Host-side unit tests for `pid.c`, `imu_filter.c`, and the
parameter-store CRC, run in CI.

## Constraints

- **Memory**: 520 KB SRAM, 2 MB flash. Note: the pico-sdk board header
  claims 4 MB — the parameter sector is pinned at 2 MB − 4 KB, valid
  either way. Firmware + param sector must fit in 2 MB (CI-enforced).
- **Timing**: control loop period 2 ms ± jitter budget 100 µs; the
  I2C burst read (~0.4 ms @ 400 kHz) fits inside the tick.
- **Tuning is tethered**: USB CDC only; the robot balances untethered
  but is tuned on a leash.
- **Torque**: no closed-loop torque feedback — steppers are commanded
  by acceleration with clamps chosen so they never skip in normal
  operation.
- **Power**: motors on battery VMOT only; USB and VMOT grounds common;
  drivers current-limited via Vref before first arm.

## Acceptance Criteria

- [ ] CI builds `balancebot.uf2` inside the pinned toolchain on every
  PR touching the project.
- [ ] Streamed pitch angle tracks a protractor tilt within ±2° and
  shows < 0.1° RMS noise at rest, no visible drift over 60 s.
- [ ] Open-loop `rate` command spins each wheel smoothly from 30 to
  4000 steps/s with clean direction reversals.
- [ ] Robot balances ≥ 60 s untethered and recovers from a light push.
- [ ] Tilt cutoff disables both drivers within 100 ms of |pitch|
  crossing 45°.
- [ ] `save`, power cycle, `get` shows the tuned gains restored.
- [ ] Host unit tests for PID / filter / param CRC pass in CI.

## Implementation Order

1. Track A — Build infrastructure + scaffold (Phase 1)
2. Track B — IMU pipeline (Phase 2) — depends on A
3. Track C — Stepper drive (Phase 3) — depends on A, parallel with B
4. Track D — Closed-loop balancing (Phase 4) — depends on B + C
5. Track E — Tuning shell, persistence, velocity loop (Phase 5) —
   depends on D

Tracks B and C touch disjoint hardware and can be bench-verified
independently before Track D closes the loop.
